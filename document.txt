======================================================================
1. SYSTEM OVERVIEW
======================================================================
ConsoliData is a role-based procurement workflow system for the University
of Northern Philippines. The system is built with Laravel 12, PHP 8.2,
Blade templates, MySQL-style database tables, and browser-based printing.

Main roles:
1. Admin
   - Validates PPMP
   - Validates Purchase Request
   - Manages users
   - Views consolidated reports
   - Prints formal documents
2. Unit / College User
   - Creates and uploads PPMP
   - Adds PPMP items
   - Creates Purchase Requests from approved PPMP items
   - Confirms approved Purchase Requests before BAC processing
   - Views notifications, status, and remaining balances
3. BAC
   - Reviews confirmed requests
   - Uploads award/failure notices
   - Consolidates requests
   - Records bidding, awarding, delivery, and inventory

Important note for defense:
- This project is mainly a Laravel web application using Blade views and
  form submissions.
- It is not an API-first SPA.
- There is no separate public REST API layer in routes/api.php in the
  current repository snapshot.

5. DATABASE TABLES AND RELATIONSHIPS
======================================================================

Core tables:
1. users
   - login account
   - role
   - department_unit_id
   - fund_source_id
   - status
   - must_change_password

2. ppmps
   - PPMP header/master
   - belongs to a user
   - linked to department and fund source

3. ppmp_items
   - items inside a PPMP
   - stores quantity, unit, unit_cost, estimated_budget, quarter data

4. purchase_requests
   - PR header/master
   - linked to user and PPMP
   - stores purpose, status, submitted_at, confirmed_at, review fields

5. purchase_request_items
   - items inside a PR
   - linked to source ppmp_item_id for traceability

6. notifications
   - in-app notification table

7. department_units
   - lookup for colleges/offices

8. fund_sources
   - lookup for available fund sources

9. categories
   - item categories

10. preset_items
    - master item list used in PPMP entry

BAC tables:
11. consolidated_items
12. consolidated_item_sources
13. biddings
14. deliveries
15. inventories

Main relationship explanation:
- One user can have many PPMPs.
- One PPMP can have many PPMP items.
- One PPMP can be the source of many Purchase Requests.
- One Purchase Request can have many Purchase Request items.
- Each Purchase Request item can point back to one PPMP item.
- Confirmed PR items can be merged into consolidated_items.
- Awarded items can proceed to deliveries and then inventories.

======================================================================
6. MAIN WORKFLOW SUMMARY
======================================================================

6.1 Authentication flow
1. User picks role from landing page.
2. User logs in on role-specific page.
3. System validates credentials.
4. Unit users must also choose the correct fund source.
5. Session is regenerated.
6. User is redirected by role.

6.2 PPMP flow
1. Unit creates or uploads PPMP.
2. PPMP starts as Draft or Submitted.
3. Unit adds PPMP items manually or from preset items.
4. Admin reviews PPMP.
5. Admin approves or returns PPMP to draft.
6. Approved PPMP becomes source for Purchase Requests.

6.3 Purchase Request flow
1. Unit creates PR from approved PPMP items.
2. System checks remaining quantity and remaining budget.
3. PR starts as Draft.
4. Unit submits PR to admin.
5. Admin approves or disapproves.
6. If Approved, the unit must confirm.
7. Only after confirmation can BAC continue processing.

6.4 BAC procurement flow
1. BAC sees approved/confirmed requests.
2. BAC uploads failure or award notice.
3. BAC consolidates confirmed requests.
4. BAC stores bids and awards a supplier.
5. BAC records deliveries.
6. System updates inventory.

Status lifecycle for PR:
Draft -> Submitted -> Approved -> Confirmed -> Completed
Alternative failure path:
Submitted -> Disapproved

7. SECURITY AND VALIDATION FEATURES
======================================================================

1. Role-based middleware
   - Admin, Unit, BAC cannot access each other's areas.

2. Session regeneration
   - On successful login.

3. Logout invalidation
   - Session invalidated and token regenerated on logout.

4. Secure landing page behavior
   - Returning to role selection logs out the current user.

5. No-cache protection
   - Prevents back-button reuse of protected pages.

6. Three failed login attempts lock
   - The system sends an email after repeated failed attempts.

7. Signed password reset choice link
   - Reset link is time-limited and tied to current password hash.

8. must_change_password flag
   - Temporary passwords force the user to change password.

9. Unit fund-source validation
   - Unit must choose the fund source that matches the account.

10. Quantity and budget validation
    - PR cannot exceed remaining approved PPMP balance.

11. Custom in-system confirmation modal
    - Replaces browser confirm() popups.

12. 419 session-expired recovery
    - Redirects back safely with proper message.

8. POSSIBLE DEFENSE QUESTIONS AND SAMPLE ANSWERS
======================================================================

1. What kind of system is this?
   Answer:
   It is a Laravel 12 web-based procurement workflow system with three
   role-based portals: Admin, Unit, and BAC.

2. Is this system API-first?
   Answer:
   No. It mainly uses Blade views and web routes. It only has small helper
   JSON endpoints like the fund source lookup.

3. What framework did you use and why?
   Answer:
   I used Laravel because it already provides routing, authentication,
   validation, ORM, mailer, middleware, and secure session handling.

4. What are the main modules of the system?
   Answer:
   Authentication, PPMP management, Purchase Request management, Admin
   validation, BAC procurement flow, notifications, printing, and reporting.

5. What is the purpose of PPMP in your system?
   Answer:
   PPMP is the procurement planning document. It defines planned items,
   quantities, budget, and approved source data before a Purchase Request
   can be created.

6. Why did you separate Admin, Unit, and BAC?
   Answer:
   Because each role has different responsibilities and permissions, so the
   code uses role-based middleware and separate route groups.

7. How do you stop a unit from seeing another unit's records?
   Answer:
   Queries in UnitController and Unit/PpmpController are filtered using the
   currently authenticated user ID.

8. How do you validate the unit fund source during login?
   Answer:
   The selected fund_source_id is checked against the user's assigned
   fund_source_id and department_unit_id in AuthController.

9. How do you handle password recovery?
   Answer:
   The system sends a temporary password and a signed reset-choice link.
   The user can reset immediately or log in with the temporary password and
   change it inside the system.

10. Why use a signed reset link?
    Answer:
    It adds security by expiring after a set time and becoming invalid once
    the password changes.

11. How is brute-force login handled?
    Answer:
    The system counts failed attempts in session, locks after three failures,
    and sends a login-attempt alert email.

12. How does the admin review PPMP?
    Answer:
    Admin opens the PPMP validation module, reviews the submitted PPMP and
    items, then approves it or returns it to draft with remarks.

13. How does the admin review Purchase Requests?
    Answer:
    Admin checks submitted PRs, validates item details and purpose, then
    approves or disapproves with review remarks.

14. Why is there a Confirmed status after Approved?
    Answer:
    Because the approved request must still be acknowledged by the unit
    before BAC can process it.

15. How do you prevent a PR from exceeding PPMP balance?
    Answer:
    The system computes remaining quantity and remaining budget using
    remainingMetrics() before submission and confirmation.

16. How is traceability maintained between PPMP and PR?
    Answer:
    purchase_requests stores ppmp_id and purchase_request_items stores
    ppmp_item_id so each requested item can be traced back to its source.

17. How does BAC consolidation work?
    Answer:
    Confirmed request items are grouped into consolidated items, and their
    source links are stored in consolidated_item_sources.

18. How does bidding work in the system?
    Answer:
    BAC stores supplier bids, then selects a winner. The winning bid is
    marked Won while other bids can be marked Lost.

19. How is delivery connected to inventory?
    Answer:
    Once BAC records a delivery, the system creates or updates inventory
    records using the delivered quantity.

20. How are notices handled?
    Answer:
    BAC uploads award or failure notice files to the public storage disk,
    and the file path is saved on the purchase request.

21. How is printing implemented?
    Answer:
    The system uses dedicated print Blade views and browser-based
    window.print() instead of a PDF package.

22. Why did you use both Eloquent and DB query builder?
    Answer:
    Eloquent is used for standard entity operations, while DB query builder
    is better for grouped summaries, joins, and raw aggregates.

23. Why did you use a custom Notification model instead of Laravel
    notification classes?
    Answer:
    The current implementation uses a simple database-backed custom model,
    which was easier to control for the current workflow.

24. Why are emails wrapped in try/catch?
    Answer:
    Because email failure should not stop the business process. The database
    notification remains the reliable audit trail.

25. How is CSV import implemented?
    Answer:
    The system uses fopen() and fgetcsv() to read uploaded PPMP CSV files,
    map columns, validate values, and create PPMP item records.

26. What is one important security feature of the system?
    Answer:
    The no-cache and session invalidation logic prevents the user from going
    back to protected pages after logout using the browser back button.

27. What is the most important relationship in the database?
    Answer:
    The link between PPMP, PPMP items, Purchase Request, and Purchase
    Request items because it preserves planning-to-request traceability.

28. Why does the system use review_remarks?
    Answer:
    It allows the reviewer to explain why a PPMP or PR was rejected or
    returned, which helps the user correct the document.

29. What should you say if asked whether there is a mobile app?
    Answer:
    This is a responsive web system, not a separate mobile application.

30. What should you say if asked whether this system uses external payment
    or procurement APIs?
    Answer:
    It does not currently use external procurement/payment APIs. It is an
    internal workflow and document management system.

9. POSSIBLE PANEL QUESTIONS ABOUT CODE QUALITY OR LIMITATIONS
======================================================================

1. Why are there missing create-migrations for some tables?
   Suggested answer:
   The current repository snapshot does not show all create-migrations, but
   the live database already contains the needed tables. The code still
   documents and uses those tables consistently.

2. Why no queue for emails?
   Suggested answer:
   The current version sends emails synchronously for simplicity, but it can
   be improved later by moving mails to queued jobs.

3. Why no PDF generation?
   Suggested answer:
   The current requirement was satisfied using browser print views for fast
   deployment and easy layout control.

4. Why no policies/gates?
   Suggested answer:
   Access control is currently enforced mainly by route middleware and
   controller ownership checks. Policies can be added later as a refactor.

5. Why mix some logic in Blade and controllers?
   Suggested answer:
   Most business logic is in controllers, but some dashboard display metrics
   and print formatting remain in Blade for direct presentation.

6. Are all status labels consistent everywhere?
   Suggested answer:
   The main PR workflow is Draft -> Submitted -> Approved -> Confirmed ->
   Completed, but some old labels may still appear in a few views and can be
   standardized further.
